Introducción

RetroAPI proporciona datos en formato JSON como un servicio de estilo REST que permite a los desarrolladores explorar e integrar datos de ordenadores y consolas retro, sus juegos y emuladores en sus proyectos. Además, permite añadir, modificar y borrar esos datos para ampliar la información que contiene (en un futuro esta posibilidad será solo accesible a usuarios registrados).

Inicio rápido

Las API impulsan gran parte del software que usamos hoy, desde las aplicaciones en nuestros teléfonos y relojes hasta la tecnología que vemos en los deportes y programas de televisión. RetroAPI se ha creado para permitir que cualquier persona obtenga o modifique sus datos de forma continua y sencilla.

Por ejemplo, se puede acceder a todos los dispositivos que almacena visitando la siguiente URL:

https://retroapi-daw.herokuapp.com/api/v1/devices?all=1

Para consultar un dispositivo específico, en lugar de toda la lista:

https://retroapi-daw.herokuapp.com/api/v1/devices?name=Super Nintendo

Métodos de petición HTTP

En los ejemplos anteriores se usó como método de petición GET, que se emplea para recuperar datos. Sin embargo, RetroAPI también utiliza otros tres métodos: POST, PUT y DELETE. La funcionalidad de estos métodos viene marcada por la metodología REST, y es la siguiente:

GET: Como se ha dicho, sirve para recuperar datos. Cualquier petición que se haga usando este método tendrá como respuesta un JSON con uno o varios documentos (dispositivos, juegos o emuladores).

POST: Se usa para añadir datos nuevos a cualquiera de las colecciones.

PUT: Este método se emplea para indicar que se quiere modificar un documento ya existente.

DELETE: Por último, el método DELETE borrará un documento (siempre que no esté referenciado en otra colección).

Endpoints

Los endpoints son las URLs de una API o un backend que responden a una petición. En el caso de RetroAPI existen tres endpoints que se corresponden con los tres recursos o colecciones de datos a los que se puede acceder; esto es: devices (dispositivos), games (juegos) y emulators (emuladores).

Por tanto, para acceder a cualquiera de estas colecciones y realizar la operación que se desee con ellas, es necesario añadir a la URL base (https://retroapi-daw.herokuapp.com) el endpoint que indicará el recurso elegido.

Por ejemplo, si se desea añadir un nuevo documento a la colección de emuladores, se utilizaría el método POST con la siguiente URL: 

https://retroapi-daw.herokuapp.com/emulators

Con esto, RetroAPI sabrá qué se quiere hacer y dónde se quiere hacer. Sin embargo, aún es necesario un paso más: indicar cuál es el documento que se quiere añadir. Para ello se deberá incluir en el cuerpo (body) de la petición un JSON con los datos del nuevo emulador. Un ejemplo sería: 

                                
{     
    "name":"RetroArch",
    "web": "https://www.retroarch.com/",
    "license": "GPLv3",
    "author": "The Libreto Team",
    "description": [
        {
            "lang": "es",
            "content": "RetroArch es un programa con implementación de la API de libretro, diseñado..."
        }
    ]
}

En el ejemplo se añadiría a la colección de emuladores RetroArch. No todas las propiedades son necesarias para crear un nuevo documento, como veremos a continuación.

Parámetros

Cada endpoint tiene una serie de parámetros que pueden hacer referencia a un campo de la base de datos o suministrar información a RetroAPI de cómo se quiere recibir la información: un único documento, toda la colección, el idioma, etc.

Dispositivos

Tabla de parámetros de un dispositivo
Parámetro Función
all=1 (solo GET) Se enviará un JSON conteniendo un array con todos los documentos de la colección de dispositivos. Este parámetro debe ser el único en la petición GET. Si se añade algún parámetro más (excepto lang), o se asigna cualquier valor distinto de 1, se devolverá un error.
like=1 (solo GET) Si a una petición GET se le añade este parámetro, se buscarán los campos señalados que contengan la cadena indicada. Por ejemplo, si se quiere una lista con todos los dispositivos que contengan la palabra "Nintendo", se haría la siguiente petición: https://retroapi-daw.herokuapp.com/devices?name=Nintendo&like=1 Sin el parámetro "like", solo se devolvería el dispositivo cuyo nombre fuera exactamente "Nintendo" (en caso de haberlo).
name String. El nombre del dispositivo. Es el único campo obligatorio cuando se añade, modifica o borra un dispositivo.
image Array de strings. Las URLs de las imágenes del dispositivo.
description String. La descripción del dispositivo.
lang String. La lengua de la descripción. Si no se añade a la URL, por defecto se devolverá la descripcion en inglés. Se puede añadir a cualquier otro parámetro de búsqueda, incluido all.
manufacturer String. La empresa que fabricó el dispositivo.
type String. El tipo de dispositivo: ordenador, consola, etc.
year Númerico. El año de fabricación del dispositivo.
architecture String. La arquitectura del dispositivo: 8 bits, 16 bits, etc.
cpu String. La CPU del dispositivo.
memory String. La cantidad de memoria RAM del dispositivo.
gamepad String. Una URL que apunte a la imagen del mando del dispositivo.
games Array de documentos de tipo game. Un array que contiene los juegos del dispositivo. Para añadir, modificar o borrar un juego, solo es necesario indicar su nombre en el cuerpo de la petición POST, PUT o DELETE.
emulators Array de documentos de tipo emulator. Un array que contiene los emuladores del dispositivo. Para añadir, modificar o borrar un emulador, solo es necesario indicar su nombre en el cuerpo de la petición POST, PUT o DELETE.

Cualquier otro parámetro que no esté en la tabla de arriba, devolverá un error en la petición.
